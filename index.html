Controladores de Versão:

É um sistema que registra as mudanças feitas em um arquivo ou um conjunto de arquivos ao longo do tempo de forma que você possa recuperar versões específicas.

O que é?

Subversion (SVN):

é um sistema de controle de versão centralizado que gerencia arquivos e diretórios controlando as alterações realizadas ao longo do tempo. Além disso, é possível recuperar versões anteriores ou visualizar o histórico de alterações. Tem como ponto forte a utilização em rede, possibilitando que vários usuários possam trabalhar colaborativamente.

Permite que você recupere versões antigas de seus dados, ou que examine o histórico de suas alterações. É um sistema de caráter geral que pode ser usado para gerenciar quaisquer conjuntos de arquivos (código-fonte, arquivos de edição de vídeo, etc.).

GIT:

É um sistema de controle de versão distribuído. Atráves deles podemos desenvolver projetos na qual diversas pessoas podem contribuir simultaneamente no mesmo, editando e criando novos arquivos e permitindo que os mesmos possam existir sem o risco de suas alterações serem sobrescritas.

Curiosidade:

Subversion (SVN):

O subversion foi criado a partir das decorrências dos bugs e limitações do Concurrent Version System (CVS), assim a CollabNet optou por iniciar a construção do zero. Devido a isso, SVN começou a ser utilizada pelos os desenvolvedores para gerir seu próprio código-fonte em 31 de agosto de 2001.

O nome Subversion foi originalmente criado por Jim Bland, que junto com Karl Fogel, Collins-Sussman, Brian Behlendorf e Jason Robbins formaram o primeiro grupo de desenvolvedores do novo projeto.

Um marco importante para o SVN foi a integração com o Apache Software Foundation (ASF), que iniciou em 2009. No início de 2010, o Subversion foi totalmente adotado na família do ASF de projetos de nível superior, transferindo o projeto para http://subversion.apache.org, e foi rebatizado de "Apache Subversion".

GIT:

O desevolvimento do GIT teve inicio quando vários desenvolvedores do Kernal Linux resolveram deixar de usar o BitKeeper. Linus e a comunidade de desenvolvedores do Linux não ficaram satisfeitos com nenhuma das ferramentas que trabalhavam de forma descentralizada disponíveis naquela época, e optaram então por desenvolver a sua própria ferramenta de controle de versões.

Linus dizia que seria importante que essa nova ferramenta não trabalhasse de forma semelhante ao CVS, segundo ele, um exemplo de software mal projetado. Linus também afirmou que era impossível desenvolver um CVS de forma correta.

Outro ponto importante sobre o que deu origem ao GIT, é que a Linus declarou que a ferramenta deveria ser semelhante ao BitKeeper, no sentido de trabalhar de forma distribuída. Alto desempenhos e medidas de segurança contra corrompimento de arquivos também eram necessários.

Linus comandou o desenvolvimento do GIT até ele começar a ser utilizado por outros usuários com o conhecimento técnico avançado. A partir disso, Junio Humano assumiu a gerência do projeto.

Ciclo Básico de Trabalho:

Subversion (SVN):

• Atualizar cópia de trabalho: através do comando svn update, que traz as alterações de repositório para sua cópia de trabalho.

• Realizar alterações: através dos comandos svn add, que adiciona arquivos, diretórios ou links simbólicos, svn delete, que exclui um item de uma cópia de trabalho ou do repositório, svn move, que move um arquivo ou diretório.

• Verificar alterações: através dos comandos svn status, que exibe informações sobre o estado de arquivos e diretórios na cópia de trabalho e svn diff, que exibe as diferenças entre duas revisões ou caminho.

• Possivelmente desfazer algumas alterações: com o comando syn revert, que desfaz todas as edições locais.

• Resolver conflitos: com os comandos svn update, que atualiza sai cópia de trabalho e svn resolved, que remove arquivos ou diretórios da cópia de trabalho do estado de “conflito”.

GIT:

• Atualizar seu repositório e cópia de trabalho: através do comando de git fetch, que faz o download de alterações do repositório desejado, seguido de um comando git marge, para mesclar al alterações baixadas com o seu repositório, ou através de comando git pull, que atualiza seu repositório e sua área de trabalho com as alterações de outro repositório.

• Fazer alterações: através dos comandos git add, que adiciona arquivos ao index, git rm, que exclui um item de uma cópia de trabalho ao do repositório e git mv, que move ou renomeia um arquivo ou diretório.

• Verificar suas alterações: através dos comandos git status, que exibe informações sobre o estado de arquivos e diretórios na cópia de trabalho e git diff, que exibe as diferenças entre duas revisões ou caminhos.

• Possivelmente desfazer algumas alterações: com o comando git revert, que desfaz todas as edições locais.

• Resolver conflitos: com o comando git mergetool, que executa ferramenta de resolução de conflitos para resolver conflitos de junção.

• Submeter alterações: através do comando git commit¸ que envia as alterações de sua cópia trabalho para o repositório.

• Possivelmente propagar suas alterações para outro repositório: através do comando git push, que envia as alterações do repositório local (origem) para outro repositório (destino).

Licença:

Subversion (SVN):

O Subversion antes seguia os termos da Debian Free Software Guidelines (DFSG), mas com apoio da Apache Software Foudation (ASF), passou a adotar os termos da licença Apache.

Contudo, é uma licença de software livre de autoria da ASF, onde ela requer preservação do aviso de direitos autorais e aviso legal, porém, permite o uso de códigos fonte para o desenvolvimento de software proprietário, bem como software livre e open source.

GIT:

O GIT segue os termos de licença General Public License (GPL). A GPL foi criada por Richard Stallman em 27 de setembro de 1983. De acordo com Richard Stallman a principal mudança realizada na segunda versão da licença GPL é a inclusão de uma cláusula chamada por ele de “liberdade ou Morte”. Essa cláusula define que ninguém pode impor restrições a terceiros de distribuir seu software, restringindo a distribuição em forma binária, por exemplo.

Características:

Subversion(SVN):

• Consegue controlar as alterações realizar na cópia de trabalho que ainda não foram enviadas ao repositório, através do comando svn diff.

• Todo commit no Subversion pode ser acompanhado de uma mensagem de log, ou seja, o usuário pode descrever todas as alterações relativas aquele commit em uma mensagem, que fica gravado no repositório. Mas o subversion também permite o commit parcial, ou seja, o usuário tem a opção de realizar commit selecionado apenas o diretório desejado.

• O Subversion permite apenas uma mensagem de log para todo o conjunto de mudança, não permitindo que cada arquivo possua uma mensagem de log particular.

• O Subversion oferece ao usuário a opção de não somente realizar um check out no repositório como um todo, mas também um check out parcial, ou seja, o usuário tem a opção de realizar um diretório.

GIT:

• Com o GIT é possível controlar as alterações realizadas na cópia de trabalho que ainda não foram enviadas ao repositório. Isso é feito através do comando git diff, assim o usuário pode analisar suas alterações com cautela antes de enviá-las ao repositório.

• É possível também ignorar alterações indesejadas, alocando-as em um diretório isolado, através do comando git stash.

• Como no Subversion, no GIT todo commit pode ser acompanhado de uma mensagem de log, ou seja, o usuário pode descrever todas as alterações relativas aquele commit em uma mensagem, que fica gravada no repositório.

• O GIT também permite apenas uma mensagem de log para todo o conjunto de mudanças, não permitindo que cada arquivo do conjunto de mudanças possua uma mensagem de log particular.

Interface:

Subversion (SVN):

O Subversion é uma ferramenta de controle de versões com maior número de clientes baseados em interface web. Geralmente, esses clientes possibilitam que o usuário navegue através de árvores e das várias revisões dos arquivos, além de possibilitar a execução de diffs, o trabalho com o formato rss, a execução de check outs, entres outras funcionalidades. São exemplos de clientes de interface web para o Subversion as ferramentas ViewVC, SVN::Web, WebSVN, ViewSVN, mod_svn_view, Chora, Trac, SVN::RaWeb::Light, SVN Browser, Insurrection e perl_svn.

Existem também muitos clientes que trabalham em modo gráfico, ou Graphical User Interface (GUIs), para Subversion. São exemplos de GUIs para Subversion as ferramentas RapidSVN, que é multiplataforma, TortoiseSVN, que funciona como um pluing in para Windows Explorer e Jsvn, desenvolvido com API Java para interface gráficas Swing.

GIT:

Apesar de o GIT ter sido desenvolvido visando um bom desempenho para o sistema operacional Linux, e muitos usuários de Linux preferiam usá-lo através de uma linha de comando, existem bons clientes que trabalham em modo gráfico, ou GUIs, disponíveis para a ferramenta. O cliente Gitk já acompanha a distribuições do GIT. Estão disponíveis também, bons clientes como o Qgit, Git-guie o TortoiseGit, que foi desenvolvido para usuários do Windows.

O GIT não possui clientes baseados em interface em web como o Subversion, porém sua distribuição já possui em cliente, o Gitweb. O Gitweb foi desenvolvido na linguagem Perl e permite navegar pelos diretórios através de revisão desejadas, ver o conteúdo dos arquivos, verificar o log de diferentes ramos e examinar os commit. Além disso, ele trabalha com feeds em rss ou no formato atom. Outro bom cliente de interface web para o GIT é o Cgit, que possui uma interface melhor do que a do gitweb e disponibiliza ainda mais detalhes.

Vantagens e Desvantagens:

Subversion (SVN):

Vantagens:

• Permite travar arquivos impedindo atualizações.
• Controle de privilégios de acesso mais sofisticados. • Commits atômicos.
• Conceito de revisão.

Desvantagens:

• Depedente de um servidor. Em caso de pane no servidor é impossível fazer um update ou commit.
• A cada commit é necessário se conectar ao servidor o que torna o sistema lento.
• Caso um arquivo seja renomeado seu histórico de alterações é perdido.
• Difícil de se usar com grandes equipes ou equipes remotas.

GIT:

Vantagens:

• Em quase tudo é executado mais rápido, em alguns casos muito mais rápido por ser otimizado para funcionar pela internet.
• É muito mais fácil administrar diversas fontes de atualizações.
• Permite trabalhar confortavelmente sem perder nenhuma funcionalidade e informação sem estar conectado ao servidor central (que pode e frequentemente é usado). Ele tem mais metadados locais.
• A história do projeto pode ser modificada. • Possui uma staging area que permite selecionar partes que deseja enviar para um repositório.

Desvantagens:

• GIT não oferece ao usuário a opção de realizar um check out principal.
• Curva de aprendizado média.
• Para quem prefere ferramenta gráfica pra tudo, deixa a desejar.
